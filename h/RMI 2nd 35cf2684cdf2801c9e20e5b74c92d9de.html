🐍 PYTHON VERSION — Pyro4

What is Pyro4?

Pyro = Python Remote Objects
Unlike XMLRPC (which sends XML text), Pyro4 sends actual Python objects over the network. It feels even more natural — like the remote object is sitting right in your code.

XMLRPC  → converts everything to XML text
Pyro4   → sends Python objects directly (pickled)

🖥️ SERVER SIDE (Python/Pyro4)

import Pyro4

Imports the Pyro4 library — handles all network communication automatically.

@Pyro4.expose
class StringConcatenator:
  • class StringConcatenator → a normal Python class
  • @Pyro4.expose → a decorator that says:
    "Make ALL methods in this class callable by remote clients"
    • Without this → methods are private (local only)
    • With this → methods are exposed (clients can call them over network)
    def concatenate(self, str1, str2):
        print("string 1: ", str1)
        print("string 2: ", str2)
        return str1 + str2
  • self → refers to the object itself (standard Python)
  • str1, str2 → two strings received from the client
  • print(...) → prints on the server's screen (not client's!)
  • return str1 + str2 → joins strings and sends result back
    • "Hello " + "World" → "Hello World"
def start_server():
    concatenator = StringConcatenator()
  • Creates an instance (object) of the class
  • This is the actual object that will serve client requests
    daemon = Pyro4.Daemon()
  • Daemon → the heart of the Pyro4 server
  • It runs in the background, listening for incoming client calls
  • Think of it as the receptionist who receives calls and routes them
Client Call ──► Daemon (receptionist) ──► concatenator object ──► returns result
    uri = daemon.register(concatenator)
  • register() → gives the object a unique address on the network
  • uri = Uniform Resource Identifier (like a URL for the object)
  • Example URI: PYRO:obj_abc123@localhost:56789
PartMeaning
PYRO:Protocol (like http://)
obj_abc123Unique ID of the object
localhostServer machine
56789Port number (auto-assigned)
    print("Server URI:", uri)
    print("Server is ready...")
    daemon.requestLoop()
  • Prints the URI — client needs this to connect!
  • requestLoop() → keeps server running forever, handling requests
    • Like serve_forever() in XMLRPC
if __name__ == "__main__":
    start_server()
  • Only runs start_server() if this file is run directly
  • Prevents it from running if this file is imported elsewhere
  • Best practice for Python programs

💻 CLIENT SIDE (Python/Pyro4)

import Pyro4

Same library, but used differently on client side.

uri = input("Enter Server URI: ")
  • User must type the URI that the server printed
  • e.g., PYRO:obj_abc123@localhost:56789
  • This is how the client knows where the server is
server = Pyro4.Proxy(uri)
  • Proxy → creates a fake local object that represents the remote server
  • Every method you call on server → secretly travels over network to real server
  • Looks like: server.concatenate() but works like a network call
server.concatenate("Hi", "Bye")
        │
        │  [Pyro4 magic]
        │
        ▼
  Network → real StringConcatenator.concatenate("Hi", "Bye") on server
        │
        ▼
  Returns "HiBye" back to client
s1 = input("Enter first string: ")
s2 = input("Enter second string: ")
result = server.concatenate(s1, s2)
print("Concatenated String:", result)
  • Gets two strings from user
  • Calls remote method (looks local, works remote!)
  • Prints the result returned by server

☕ JAVA VERSION — RMI

What is Java RMI?

RMI = Remote Method Invocation
Java's built-in version of RPC. Works the same idea — call a method on a remote machine — but uses Java's type system and interfaces.

Python Pyro4  ≈  Java RMI
(same concept, different language)

📋 The Interface — StringIface

Before looking at client, understand this: Java RMI requires an interface that both client and server agree on.

// StringIface.java (not shown but implied)
import java.rmi.*;
public interface StringIface extends Remote {
    String concat(String s1, String s2) throws RemoteException;
}
  • interface → a contract/agreement: "I promise to have these methods"
  • extends Remote → marks it as a remotely callable interface
  • throws RemoteException → network calls can fail, so this is mandatory
  • Both client and server use this same interface — ensures they agree on method names

🖥️ SERVER SIDE (Java RMI) — implied

The server would:

  1. Implement StringIface
  1. Register the object with name "string" on port 1337
  1. Wait for client calls

💻 CLIENT VERSION 1 — Hardcoded Strings

import java.rmi.*;

Imports Java's built-in RMI package (no external library needed — it's part of Java).

public class Client {
    public static void main(String[] args) {
  • public class Client → class name must match filename Client.java
  • public static void main(String[] args) → entry point of every Java program
        try {

Everything inside try { } is protected — if anything goes wrong (network error, server down), it jumps to catch.

            String URI = "rmi://127.0.0.1:1337/string";

The server's address in RMI format:

PartMeaning
rmi://Protocol (like http://)
127.0.0.1IP of server (127.0.0.1 = localhost = this machine)
1337Port number
/stringName the server registered the object under
            StringIface serv = (StringIface) Naming.lookup(URI);

This is the most important line — breaks down into parts:

  • Naming.lookup(URI) → contacts the server at that URI and fetches the remote object
  • (StringIface) → cast — tells Java "treat this as a StringIface object"
  • StringIface serv → stores it in variable serv

Now serv looks like a local Java object, but every method call goes over the network!

Naming.lookup("rmi://127.0.0.1:1337/string")
         │
         ▼
  Contacts RMI Registry at 127.0.0.1:1337
         │
         ▼
  Finds object registered as "string"
         │
         ▼
  Returns a proxy/stub object → stored in serv
            String str1 = "Hello ";
            String str2 = "World";

Hardcoded strings — no user input in Version 1.

            String result = serv.concat(str1, str2);
  • Calls concat() on the remote server object
  • Looks like a normal Java method call
  • Secretly: packages args → sends over network → server runs concat() → returns result
            System.out.println("Concatenated String: " + result);
  • System.out.println() → Java's print statement (like Python's print())
  • Prints: Concatenated String: Hello World
        }
        catch (Exception e) {
            System.out.println(e.getMessage());
            e.printStackTrace();
        }
  • catch (Exception e) → catches any error that happens
  • e.getMessage() → prints a short error description
  • e.printStackTrace() → prints the full error trail (for debugging)

Common errors caught here:

Server not running      → ConnectException
Wrong URI               → NotBoundException
Network issue           → RemoteException

💻 CLIENT VERSION 2 — User Input with Scanner

Everything is the same as Version 1 except the string input part:

            Scanner sc = new Scanner(System.in);
  • Scanner → Java's tool for reading user input
  • System.in → the keyboard (standard input stream)
  • Together: "Create a scanner that reads from the keyboard"
            System.out.print("Enter first string: ");
            String str1 = sc.nextLine();
  • System.out.print() → prints without newline (cursor stays on same line)
  • sc.nextLine() → reads a full line of text the user types (until Enter)
  • String str1 → stores the input
Console looks like:
Enter first string: Hello World_
                    ↑ user types here
            System.out.print("Enter second string: ");
            String str2 = sc.nextLine();

Same — reads second string from user.

            String result = serv.concat(str1, str2);
            System.out.println("Concatenated String: " + result);
            sc.close();
  • Same remote call as Version 1
  • sc.close() → closes the Scanner (frees memory — good practice!)
